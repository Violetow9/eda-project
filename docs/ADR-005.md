# ADR-005 : Audit trail des actions Task et Project

---

## Statut

Accepté

---

## Contexte

TaskFlow doit permettre de répondre à la question :

> Qui a fait quoi, sur quelle entité, et à quelle heure ?

Toute action d'écriture sur les entités `Task` et `Project` doit être tracée automatiquement.

Cette traçabilité ne doit pas polluer les services métier. Les services de gestion des tâches et des projets ne doivent pas écrire directement dans une table d'audit.

Nous voulons également pouvoir changer plus tard le backend de stockage des logs : base relationnelle, fichiers, service dédié, outil de logs spécialisé, etc.

---

## Décision

Nous avons décidé d'implémenter l'audit trail comme un module dédié qui écoute les événements métier.

Les événements audités sont notamment :

```txt
task.created
task.assigned
task.moved
task.deleted
project.created
project.deleted
member.added
```

Un `AuditListener` écoute ces événements et crée un log d'audit.

Chaque log contient :

```txt
actorId
action
entityType
entityId
occurredAt
metadata
```

Exemple :

```json
{
  "actorId": "user-1",
  "action": "task.moved",
  "entityType": "Task",
  "entityId": "12",
  "occurredAt": "2026-04-27T12:00:00.000Z",
  "metadata": {
    "projectId": 3,
    "from": "Todo",
    "to": "In Progress"
  }
}
```

Le stockage actuel est réalisé en base PostgreSQL via TypeORM.

L'accès au stockage passe par un repository d'audit afin de pouvoir remplacer l'implémentation sans modifier les services métier.

---

## Alternatives envisagées

| Alternative | Raison du rejet |
|------------|------------------|
| Écrire l'audit directement dans les services métier | Pollue la logique métier et rend le stockage difficile à remplacer |
| Utiliser uniquement les logs console | Pas suffisant pour consulter ou filtrer l'historique |
| Mettre l'audit dans les controllers REST | Ne couvre pas les autres adaptateurs entrants comme la CLI |
| Utiliser un service externe dès maintenant | Trop coûteux pour la livraison actuelle |
| Faire des triggers SQL | Couplage fort à PostgreSQL et perte du contexte métier comme `actorId` |

---

## Conséquences

**Positives :**

- L'audit est découplé de la logique métier.
- Les actions venant du REST ou de la CLI peuvent être tracées de la même manière.
- Le stockage d'audit est remplaçable.
- Les logs sont consultables via API.

**Négatives / compromis :**

- Une action qui ne publie pas d'événement ne sera pas auditée.
- Le stockage actuel en base relationnelle est simple mais pas optimisé pour un très gros volume.
- L'audit dépend de la qualité des données portées par les événements.

**Points de vigilance :**

- Chaque action d'écriture doit publier un événement métier.
- Les événements doivent contenir un `actorId`.
- Ne pas appeler directement `AuditService` depuis les services métier.
- Prévoir une stratégie d'archivage si le volume de logs augmente.
