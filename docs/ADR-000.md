# ADR-000 : Exemple — Choix du bus d'événements interne

> **Ce fichier est un exemple de référence.** Il illustre le niveau de qualité attendu pour vos ADR.
> Il ne fait pas partie de votre architecture réelle — ne le modifiez pas, créez `ADR-001.md` etc.

---

## Statut

Accepté

---

## Contexte

Dans TaskFlow, plusieurs services doivent réagir aux mêmes événements métier (ex : `task.moved`). Sans découplage, `TaskService` devrait appeler directement `NotificationService`, `WebSocketGateway`, et `AuditService`. Chaque nouvel abonné imposerait une modification du service source — violant le principe Open/Closed et rendant les tests plus complexes (chaque dépendance devrait être mockée).

Le système doit également pouvoir absorber les disruptions futures (ajout de Slack, SMS, audit...) sans modifier les services métier existants.

---

## Décision

Nous avons décidé d'utiliser le module `EventEmitter2` de NestJS comme bus d'événements interne synchrone. Les services métier publient des domain events (`TaskMovedEvent`, `MemberAddedEvent`…) sur ce bus. Les handlers (`NotificationHandler`, `WebSocketHandler`, `AuditHandler`) s'y abonnent de façon indépendante.

---

## Alternatives envisagées

| Alternative | Raison du rejet |
|-------------|-----------------|
| Appel direct entre services (`TaskService` → `NotificationService`) | Fort couplage : chaque nouvel abonné nécessite une modification du service source. Impossible d'ajouter un canal sans toucher au code métier. |
| RabbitMQ / Kafka (bus externe) | Surcharge opérationnelle injustifiée pour la phase 1. Introduit une dépendance d'infrastructure dans le domaine. À envisager si le système devient distribué. |
| NestJS `@nestjs/cqrs` (CQRS natif) | Plus verbeux pour notre besoin. Adapté à un projet avec séparation lecture/écriture stricte (CQRS complet), ce qui n'est pas notre priorité initiale. |

---

## Conséquences

**Positives :**
- `TaskService` et `ProjectService` ne connaissent aucun de leurs abonnés
- Ajouter un nouveau canal (Slack, SMS) = créer un handler, brancher sur le bus — zéro modification du domaine
- Tester `TaskService` = vérifier qu'il publie le bon event, sans mocker notification/websocket
- Retirer un handler = supprimer un fichier, rien d'autre ne change

**Négatives / compromis :**
- Les événements sont traités de façon synchrone : si un handler est lent, il bloque le thread. Acceptable en phase 1, à revisiter si les volumes augmentent.
- En cas d'exception non catchée dans un handler, elle peut remonter dans le service émetteur. Nécessite une gestion d'erreur explicite dans chaque handler (voir ADR sur la résilience).

**Points de vigilance :**
- Ne jamais injecter de dépendances infrastructure (ORM, client HTTP, etc.) directement dans les domain events
- Nommer les événements avec le format `entité.action` (`task.moved`, `member.added`) pour maintenir un ubiquitous language cohérent
- Documenter chaque nouvel événement dans le tableau du README
