# ADR-007 : Authentification JWT découplée du métier

---

## Statut

Accepté

---

## Contexte

La Disruption #1 demande de remplacer l'identifiant utilisateur simulé par une vraie authentification (inscription, connexion, token).

Exigence de durabilité explicite : une migration vers un SSO d'entreprise est envisagée dans 12 à 18 mois. Le changement de mécanisme d'authentification **ne doit pas nécessiter de modifier la logique métier** (gestion des tâches, des projets, des membres).

---

## Décision

Nous avons mis en place un module `auth/` autonome qui suit la même structure hexagonale que le reste du projet :

```txt
src/auth/
  domain/
    authenticated-user.entity.ts    ← objet métier "qui est connecté"
    token-payload.ts                ← contenu du token
    token-pair.ts                   ← couple access (+ futur refresh)
    token-provider.interface.ts     ← port de sortie
  application/
    auth.service.ts                 ← register / login
    auth.module.ts
  infrastructure/
    jwt-token-provider.ts           ← adaptateur JWT (signature/vérification)
    jwt-auth.guard.ts               ← APP_GUARD global
    roles.guard.ts                  ← APP_GUARD global
  presentation/
    auth.controller.ts              ← /v1/auth/register, /v1/auth/login
    decorators/
      public.decorator.ts
      current-user.decorator.ts
      roles.decorator.ts
```

Points clés :

- `JwtAuthGuard` est enregistré comme `APP_GUARD` global ; toutes les routes sont protégées sauf celles décorées `@Public()` (auth/register, auth/login).
- L'identité de l'appelant est extraite du token et exposée aux controllers via `@CurrentUser()` qui retourne un `AuthenticatedUser` (objet du domaine `auth/`, pas un payload JWT brut).
- Les services métier (`TaskService`, `ProjectService`) ne connaissent **pas** JWT. Ils acceptent un simple paramètre `actorId: string` dans leurs commandes ; les controllers le remplissent depuis `@CurrentUser().id`, la CLI le remplit avec `'cli'`.
- L'abstraction `TokenProvider` (port) décrit ce que le métier attend (`generate(payload)`, `verifyAccess(token)`). `JwtTokenProvider` est l'adaptateur courant.

---

## Pourquoi cette frontière tient pour un futur SSO

Le jour où l'authentification migre vers un SSO :

- Seul l'adaptateur entrant change : `auth.controller` est remplacé par un callback OAuth/OIDC, ou `JwtAuthGuard` est remplacé par un `SsoAuthGuard` qui valide un cookie de session ou un token signé par l'IdP.
- Le contrat `AuthenticatedUser { id, email, role }` reste identique côté domaine — peu importe qu'il vienne d'un JWT signé localement ou d'un IdP externe.
- Les services métier ne changent pas : ils continuent de recevoir `actorId: string` dans leurs commandes.
- `JwtTokenProvider` peut être remplacé par un autre `TokenProvider` sans toucher `AuthService`.

Aucune ligne de `TaskService` ou `ProjectService` n'aura à changer.

---

## Alternatives envisagées

| Alternative | Raison du rejet |
|-------------|------------------|
| Sessions serveur (cookie + store) | Plus de persistance/store à gérer pour la même garantie ; moins direct pour un futur SSO sans redirection |
| OAuth/OIDC dès maintenant via un IdP externe | Sur-dimensionné pour une livraison de 6h ; ajoute une dépendance opérationnelle |
| Stocker l'utilisateur courant dans un service global injecté | Casserait l'isolation requête / propagerait l'état partagé, antipattern Nest |
| Passer un header `X-User-Id` simulé (Phase 1) | Ne répond pas à la disruption qui demande une vraie authentification |
| Garder le JWT mais l'utiliser directement dans `TaskService` | Couplerait le métier au transport et empêcherait la migration SSO |

---

## Conséquences

**Positives :**

- Le métier ne connaît pas la mécanique d'authentification.
- La CLI peut exécuter les mêmes services en passant `actorId: 'cli'`.
- L'audit trail capture le bon `actorId` quel que soit l'adaptateur entrant.
- Une migration SSO se fait en remplaçant un adaptateur, pas en réécrivant le métier.

**Négatives / compromis :**

- Pas de refresh token ni de révocation pour cette livraison (TTL long en attendant).
- La gestion des rôles est binaire (`user` / `admin`) ; à enrichir si plusieurs rôles métier émergent.
- Le `JwtAuthGuard` global oblige à expliciter `@Public()` partout où l'authentification ne s'applique pas.

**Points de vigilance :**

- Ne jamais injecter `JwtService` ailleurs que dans le module `auth/`.
- Toujours passer `user.id` (et non un payload JWT) aux services métier via `actorId`.
- Le secret `JWT_ACCESS_SECRET` doit être fourni par l'environnement et tourner régulièrement en production.
