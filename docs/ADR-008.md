# ADR-008 : Temps réel via WebSocket scopé par projet, branché sur les events domaine

---

## Statut

Accepté

---

## Contexte

La Disruption #1 demande qu'un déplacement de tâche soit visible **instantanément** par les autres membres du projet, sans rechargement.

Exigence de durabilité : la technologie temps réel pourrait changer dans 6 mois (autre transport, autre infrastructure). **Ce changement ne doit pas impacter la logique de déplacement de tâche.** Les règles métier ne doivent pas être réécrites.

Exigence produit : les mises à jour temps réel sont **scopées par projet**. Un membre du projet A ne reçoit pas les events du projet B.

---

## Décision

Nous séparons strictement trois rôles :

1. **Le métier publie un event domaine** (`TaskMovedEvent`, `TaskCreatedEvent`, etc.) sans connaître l'existence du temps réel.
2. **Un listener d'infrastructure** (`ServerListener`) écoute ces events sur le bus interne (`@nestjs/event-emitter`) et délègue à la `TaskGateway`.
3. **`TaskGateway` (Socket.IO)** émet l'event vers la room `project:${projectId}` correspondante.

Le canal :

```txt
TaskService.moveTask
  -> EventPublisher.publish(new TaskMovedEvent(projectId, ...))
    -> EventEmitter2 (bus interne)
      -> ServerListener.handleTaskMoved
        -> TaskGateway.emitTaskMoved
          -> server.to('project:42').emit('task.moved', payload)
```

`TaskGateway` expose un message `project.join` qui :

- vérifie l'accès au projet via `ProjectAccessService.canAccessProject({ userId, projectId })` (port d'application, indépendant du transport) ;
- joint le client à la room `project:${projectId}` ;
- échoue avec `WsException('Forbidden project')` sinon.

---

## Pourquoi le métier reste protégé

`TaskService` ne fait référence à aucune classe Socket.IO. Il publie un `TaskMovedEvent`. Si demain :

- on remplace Socket.IO par Server-Sent Events ou WebTransport, on remplace `TaskGateway` ;
- on remplace le bus interne par Redis Pub/Sub pour multi-instances, on remplace `NestEventPublisher` ;
- on déplace le temps réel hors du processus (worker dédié), on déplace `ServerListener` et `TaskGateway` sans toucher `TaskService`.

Aucune de ces évolutions ne réécrit les règles de déplacement de tâche, ni les transitions du `TaskStatus`.

---

## Scope par projet

Plutôt que de filtrer côté client (où n'importe qui pourrait s'abonner à n'importe quel event), nous **filtrons côté serveur** via les rooms Socket.IO.

- Aucun event n'est émis sur le canal global ; chaque émission cible `to('project:${projectId}')`.
- L'adhésion à une room passe par `project.join` après vérification d'accès.
- Cette stratégie reste valide quand `ProjectAccessService` deviendra réellement strict (aujourd'hui il accepte tout pour cette livraison — voir "Points de vigilance").

---

## Alternatives envisagées

| Alternative | Raison du rejet |
|-------------|------------------|
| Polling HTTP côté front (ex. toutes les 3s) | Latence visible et coût réseau ; ne répond pas à "instantanément" |
| Server-Sent Events | Mono-directionnel, suffisant aujourd'hui mais moins flexible si on a besoin de messages client→serveur (présence, edit signal) |
| WebSocket nu sans rooms | Tout le monde reçoit tout — viole l'exigence de scope par projet |
| Faire émettre la WebSocket directement depuis `TaskService` | Couplage métier ↔ transport ; chaque changement de techno casserait le métier |
| Diffuser les events via Redis Pub/Sub dès maintenant | Sur-dimensionné pour un mono-instance ; ajout possible plus tard sans toucher le métier |

---

## Conséquences

**Positives :**

- Le métier ne connaît ni Socket.IO ni Redis ; il publie des events nommés métier.
- La sécurité de scope est centralisée côté serveur.
- L'ajout d'un nouveau type d'event (par ex. `task.commented`) se fait en publiant un nouvel event et en ajoutant une méthode `emit*` dans `TaskGateway`, sans toucher `TaskService`.

**Négatives / compromis :**

- Le temps réel dépend pour l'instant d'un **mono-instance** : la propagation entre instances backend nécessitera un adapter (Redis ou autre) sans changer le code métier.
- Les events Socket.IO ne sont pas persistés ; un client déconnecté manque les changements le temps de la reconnexion (le rechargement de la page corrige).
- L'ACL est pour l'instant permissive (`ProjectAccessService.canAccessProject` retourne toujours `true`) ; à durcir une fois la notion de membre projet finalisée côté UI.

**Points de vigilance :**

- Ne jamais émettre sur le serveur sans cibler une room de projet.
- Ne pas faire dépendre `TaskService` de `TaskGateway`.
- Quand `ProjectAccessService` deviendra strict, ne pas dupliquer la règle dans le controller HTTP — créer un port commun.
- Pour passer en multi-instance, brancher un Socket.IO adapter (Redis) **sans** modifier `TaskService`.
