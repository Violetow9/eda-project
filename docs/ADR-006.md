# ADR-006 : CLI d'administration comme adaptateur entrant

---

## Statut

Accepté

---

## Contexte

Pour faciliter les démonstrations et la génération de données de test, TaskFlow doit fournir une interface en ligne de commande permettant de :

- créer un projet ;
- créer une tâche dans un projet ;
- générer un jeu de données de démonstration.

Cette CLI ne doit pas passer par l'API REST et ne doit pas dupliquer la logique métier.

L'exigence de durabilité précise que si l'ajout d'une CLI oblige à réécrire la logique de création de projets ou de tâches, cela révèle un problème de conception.

---

## Décision

Nous avons décidé de traiter la CLI comme un adaptateur entrant, au même titre que les controllers REST.

La CLI est placée dans une couche de présentation dédiée :

```txt
src/cli/presentation
```

Elle invoque les **mêmes services applicatifs** que l'API REST :

```txt
ProjectService.create
TaskService.create
TaskService.moveTask
```

Le flux est :

```txt
CLI
  -> AdminCliCommand
    -> Service applicatif (ProjectService / TaskService)
      -> Domaine (entités + Value Objects)
      -> Repository port
      -> Infrastructure
```

La CLI ne manipule pas directement TypeORM, les repositories concrets ou les controllers REST. Elle passe un `actorId` littéral `'cli'` aux services pour que l'audit trace la source de l'action.

Le jeu de données de démonstration respecte les règles métier existantes, notamment les transitions valides du Kanban :

```txt
Todo -> In Progress -> Done
```

---

## Pourquoi pas une classe `*UseCase` séparée ?

Une première itération exposait trois classes (`CreateProjectUseCase`, `CreateTaskUseCase`, `MoveTaskUseCase`) en parallèle des services. Cela a créé deux chemins d'orchestration concurrents — un pour le REST, un pour la CLI — qui ont divergé (signatures d'events incohérentes, double publication possible).

Nous avons consolidé : un seul service par module joue le rôle de cas d'usage applicatif. Les deux adaptateurs entrants (REST et CLI) appellent les mêmes méthodes. La traçabilité du contexte (qui a fait quoi) est portée par un paramètre `actorId` sur les commandes.

Si plus tard un cas d'usage devient suffisamment complexe pour justifier une classe dédiée, il sera ajouté à côté du service sans changer cette règle d'unicité.

---

## Alternatives envisagées

| Alternative | Raison du rejet |
|------------|------------------|
| Appeler l'API REST depuis la CLI | Fonctionnel mais dépendant du transport HTTP |
| Écrire directement en base depuis la CLI | Contourne les règles métier et les événements |
| Dupliquer la logique métier dans un script | Contraire à l'exigence de durabilité |
| Utiliser un script SQL de seed | Rapide mais ne teste pas le domaine ni les use cases |
| Mettre la CLI dans les controllers REST | Mélange des adaptateurs entrants |
| Garder des classes `*UseCase` distinctes en plus des services | Doublon d'orchestration ; deux chemins divergents |

---

## Conséquences

**Positives :**

- La CLI réutilise les mêmes règles que l'API REST — un seul endroit où les events métier sont publiés.
- Les événements métier sont aussi publiés depuis la CLI ; l'audit et les notifications fonctionnent identiquement.
- Les données de démo respectent les transitions du domaine.
- L'architecture absorbe un nouvel adaptateur entrant sans duplication.

**Négatives / compromis :**

- La CLI démarre un contexte applicatif Nest, ce qui est plus lent qu'un script SQL.
- Les modules nécessaires doivent être correctement importés dans `CliModule`.

**Points de vigilance :**

- Ne pas appeler les controllers REST depuis la CLI.
- Ne pas injecter directement les repositories concrets dans la CLI.
- Garder la logique dans les services applicatifs.
- Toujours passer un `actorId` (par défaut `'cli'`) pour que l'audit reste lisible.
- Documenter les commandes dans le README.
