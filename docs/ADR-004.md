# ADR-004 : Système de notifications événementiel

---

## Statut

Accepté

---

## Contexte

Dans le cadre des évolutions produit de TaskFlow, nous devons notifier les utilisateurs lors d’événements métier importants.

Les événements concernés sont :

- `task.assigned` : notifier le membre assigné à la tâche ;
- `task.moved` : notifier les membres du projet lorsqu’une tâche est déplacée.

Les canaux demandés pour cette livraison sont :

- email ;
- notification in-app.

Chaque utilisateur doit pouvoir activer ou désactiver les canaux de son choix.

Nous devons également prévoir que d’autres canaux puissent être ajoutés plus tard, par exemple :

- Slack ;
- Teams ;
- SMS.

L’ajout d’un nouveau canal ne doit pas nécessiter de modifier les services métier de gestion des tâches ou des projets.

---

## Décision

Nous avons décidé d’implémenter les notifications comme un module séparé, déclenché par les événements métier de l’application.

Les services applicatifs comme `TaskService` ne connaissent pas directement le système de notification.

Ils publient uniquement des événements métier :

```txt
task.assigned
task.moved
```

Le module `NotificationModule` écoute ces événements via un listener dédié :

```txt
NotificationListener
```

Le flux est donc le suivant :

```txt
TaskService
  -> EventPublisher.publish('task.moved' / 'task.assigned')
    -> EventEmitter NestJS
      -> NotificationListener
        -> NotificationService
          -> EmailChannel
          -> InAppChannel
```

Cette décision permet de garder le métier découplé des détails techniques d’envoi de notification.

---

## Implémentation technique

### Bibliothèques et technologies utilisées

Pour cette livraison, nous utilisons :

| Besoin | Choix technique |
|--------|-----------------|
| Publication / écoute d’événements internes | `@nestjs/event-emitter` |
| Backend applicatif | NestJS |
| Persistance des notifications in-app | TypeORM |
| Base de données | PostgreSQL |
| Envoi email | SMTP |
| Test local des emails | Mailpit |
| Frontend | Next.js |
| Affichage temps réel / refresh notifications | Socket.IO déjà utilisé côté Kanban |

Le package `@nestjs/event-emitter` permet d’utiliser le décorateur `@OnEvent(...)` dans les listeners.

Exemple :

```ts
@OnEvent('task.assigned')
async handleTaskAssigned(event: TaskAssignedEvent): Promise<void> {
  // création des notifications
}
```

## Événements écoutés

### `task.assigned`

Lorsqu’une tâche est assignée à un utilisateur, le module notification crée une notification pour l’utilisateur assigné.

Message utilisé :

```txt
La tâche {taskId} a été assignée à {assigneeUserId}
```

Exemple :

```txt
La tâche 12 a été assignée à user-1
```

### `task.moved`

Lorsqu’une tâche est déplacée, le module notification notifie les membres du projet.

Message utilisé :

```txt
La tâche {taskId} a été déplacée par {actorId}
```

Exemple :

```txt
La tâche 12 a été déplacée par user-1
```

Le `actorId` est transmis par les cas d’usage applicatifs. Tant que l’authentification JWT n’est pas finalisée, il peut être simulé avec une valeur comme :

```txt
user-1
system
cli
```

## Canaux de notification

### Notification in-app

Les notifications in-app sont stockées en base PostgreSQL via TypeORM.

Une notification contient notamment :

```txt
id
userId
type
title
message
metadata
createdAt
readAt
```

Le champ `metadata` permet de conserver des informations utiles comme :

```json
{
  "projectId": 1,
  "taskId": 12,
  "from": "Todo",
  "to": "In Progress",
  "actorId": "user-1"
}
```

Le frontend récupère les notifications via l’API, puis filtre l’affichage par projet lorsque l’utilisateur est sur une page projet.

### Email

Le canal email est implémenté via SMTP.

En local, nous utilisons Mailpit pour capturer les emails sans les envoyer réellement à l’extérieur.

Configuration locale :

```env
SMTP_HOST=mailpit
SMTP_PORT=1025
SMTP_SECURE=false
SMTP_STARTTLS=false
SMTP_FROM=no-reply@taskflow.local
```

Mailpit expose une interface web :

```txt
http://localhost:8025
```

Cela permet de vérifier les emails envoyés pendant le développement.

En production, ce canal SMTP pourra être remplacé ou adapté pour utiliser un fournisseur externe.

## Préférences utilisateur

Chaque utilisateur peut activer ou désactiver les canaux de notification.

Les préférences sont stockées en base via TypeORM.

Exemple de préférences :

```json
{
  "userId": "user-1",
  "emailEnabled": true,
  "inAppEnabled": true
}
```

Endpoints exposés :

```http
GET /v1/notification-preferences/:userId
PUT /v1/notification-preferences/:userId
```

Payload de mise à jour :

```json
{
  "emailEnabled": true,
  "inAppEnabled": false
}
```

Le service de notification vérifie les préférences avant d’envoyer une notification.

Si `emailEnabled` vaut `false`, aucun email n’est envoyé.

Si `inAppEnabled` vaut `false`, aucune notification in-app n’est créée.

---

## API des notifications in-app

Les notifications in-app sont exposées via l’API REST.

Endpoints :

```http
GET /v1/notifications/:userId
PATCH /v1/notifications/:notificationId/read
DELETE /v1/notifications/:notificationId
```

Ces endpoints permettent de :

- récupérer les notifications d’un utilisateur ;
- marquer une notification comme lue ;
- supprimer une notification.

---

## Découplage avec le métier

Les services métier ne dépendent pas directement :

- du canal email ;
- des notifications in-app ;
- de TypeORM côté notifications ;
- de Mailpit ;
- d’un futur canal Slack, Teams ou SMS.

Ils publient seulement des événements métier.

Exemple :

```ts
this.eventPublisher.publish(
  'task.moved',
  new TaskMovedEvent(
    saved.projectId,
    saved.id,
    previousStatus,
    saved.status.toString(),
    actorId,
  ),
);
```

Le module notification est responsable de réagir à cet événement.

---

## Extensibilité des canaux

Les canaux sont représentés par une abstraction de type :

```ts
export interface NotificationChannel {
  send(input: NotificationInput): Promise<void>;
}
```

Les canaux actuels sont :

```txt
SmtpEmailChannel
InAppChannel
```

Pour ajouter Slack, Teams ou SMS, il faudra ajouter une nouvelle implémentation, par exemple :

```txt
SlackNotificationChannel
TeamsNotificationChannel
SmsNotificationChannel
```

Puis l’enregistrer dans `NotificationModule`.

Les services métier ne seront pas modifiés.

---

## Alternatives envisagées

| Alternative | Raison du rejet |
|------------|------------------|
| Envoyer les notifications directement depuis `TaskService` | Couplerait la logique métier aux canaux de notification |
| Utiliser un broker comme Kafka ou RabbitMQ dès maintenant | Trop lourd pour le besoin actuel et le temps disponible |
| Stocker uniquement les notifications en frontend | Non durable, perte des notifications au refresh et impossible à auditer |
| Envoyer uniquement des emails | Ne répond pas au besoin de notifications in-app |
| Ne pas gérer les préférences utilisateur | Ne respecte pas l’exigence produit |

---

## Conséquences

**Positives :**

- Les notifications sont découplées des services métier.
- Les canaux sont extensibles.
- Les préférences utilisateur sont centralisées.
- Les notifications in-app sont persistées.
- Les emails peuvent être testés localement avec Mailpit.
- Le système reste simple pour une première livraison.

**Négatives / compromis :**

- Les événements sont internes au processus NestJS.
- Il n’y a pas de queue ni de retry automatique en cas d’échec d’envoi email.
- Mailpit est uniquement adapté au développement local.
- Le système actuel n’est pas encore adapté à plusieurs instances backend sans mécanisme de propagation externe.

**Points de vigilance :**

- Chaque événement métier important doit être publié explicitement.
- Les préférences doivent être créées ou retournées avec des valeurs par défaut.
- Les notifications liées à un projet doivent porter un `projectId` dans leurs metadata.
- En production, l’envoi email devra être fiabilisé avec retry, monitoring et fournisseur externe.
- L’ajout de nouveaux canaux doit passer par de nouveaux adaptateurs, pas par une modification de `TaskService`.
